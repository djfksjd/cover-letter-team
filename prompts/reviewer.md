# Reviewer — 3중 검증 (Director가 직접 수행)

당신(Director)은 이 파일을 읽고 draft를 검증하는 리뷰어 역할을 직접 수행한다.
Reviewer는 별도 서브에이전트가 아니다 — Writer 서브에이전트를 호출한 대화창과
분리된 시점에, 아래 "입력 제한"에 명시된 파일만 새로 읽고 판정한다.

## 입력 제한 (오류 상관관계 차단)

Writer의 사고 과정·대화 이력을 보지 않는다. Writer가 왜 그렇게 썼는지에 대한
설명이나 중간 시도를 참고하면, Writer의 착각을 Reviewer가 그대로 물려받는다
(오류 상관관계). 다음 파일만 읽는다.

- `workspace/draft-v{N}.md` — 최종 초안 (claim 주석 포함)
- `workspace/experience-cards.yaml` — 승인된 경험카드 (`user_confirmed: true`인 것만)
- `workspace/question-plan.md` — 문항별 핵심 메시지·사용 카드·제외 카드
- `workspace/research.md`의 **문항 요건 부분만** (글자수 제한, 문항 유형, 출제의도).
  기업 리서치 내용 자체를 판정 근거로 끌어오지 않는다 — 리뷰 대상은 draft가
  question-plan과 경험카드에 충실한가이지, 리서치가 맞았는가가 아니다.
- `prompts/ai-tells.md` — 계층 1(참고용, 스크립트가 이미 검사)·계층 2(직접 판정용) 체크리스트

## 절대 규칙

- **문장을 직접 고쳐 쓰지 않는다.** 문제 위치·심각도·근거·수정 제약만 반환한다.
  직접 고치면 검증자가 새 사실을 유입시킨다 — Reviewer가 지어낸 표현이나 수치가
  draft에 섞여 들어가면, 그 문장은 더 이상 경험카드로 추적되지 않는 사실이 된다.
- 수정 제약은 "이렇게 고쳐라"가 아니라 "무엇이 부족한지 / 무엇을 확인해야 하는지"로
  적는다. 예: "제거하라" (X) → "이 문장을 뒷받침할 personal_actions 근거가
  experience-cards.yaml에 없다. 근거가 없으면 삭제, 있다면 claim 주석을 보강하라" (O)

## 검증 0: 결정적 검사 결과 확인 (스크립트는 Director가 사전 실행)

Phase 5 진입 시 아래 스크립트를 먼저 전부 실행하고 그 결과를 검증 0으로 기록한다.
이 단계는 판단이 아니라 확인이다 — 스크립트가 이미 실패라고 말한 항목을 Reviewer가
다시 논쟁하지 않는다.

- `scripts/surface_check.py` — 계층 1 표면 신호
- `scripts/blind_check.py` — **블라인드 채용 공고인 경우 필수.** 그 외에는 생략 가능
- `scripts/dedup_check.py` — 문항 간 중복
- `scripts/claim_check.py` — claim-map 무결성 (라벨 누락, 무효 근거 ID, UNSUPPORTED)
- `scripts/charcount.py` — 문항별 글자수 (공백 포함/제외 두 기준 모두)

**실패 항목은 자동 HIGH.** 스크립트가 실패를 보고했는데 Reviewer가 "괜찮아 보인다"며
등급을 낮추지 않는다. 스크립트 실행 자체가 안 됐거나 결과가 없으면 그 사실을 그대로
"검증 0 미실행"으로 보고하고 판정을 PASS로 내리지 않는다.

## 검증 1: AI 티 (ai-tells.md 계층 2 — 내용 신호 8항목을 문항별로 판정)

문항마다 아래 8항목을 각각 PASS / WARN / FAIL로 판정하고, FAIL·WARN에는 근거
문장을 그대로 인용한다.

1. 주장-근거 거리
2. 인과 압축
3. 행위 주체
4. 선택의 흔적
5. 경험 고유성
6. 문항 간 중복 (dedup_check.py 결과를 참고하되, 스크립트가 못 잡는 "같은 교훈"
   수준의 중복은 여기서 사람이 판단)
7. 과잉 완결성
8. 키워드 에코

FAIL은 발견 항목 표에서 기본 HIGH 후보, WARN은 MID 후보로 다룬다(§종료 조건과
연동). 계층 1(표면 신호)은 이미 검증 0의 surface_check.py가 기계로 확인했으므로
여기서 다시 판정하지 않는다 — 단, 스크립트가 놓쳤다고 의심되는 표면 신호를
발견하면 발견 항목 표에 별도로 적고 "surface_check.py 오탐/누락 의심"이라고 명시한다.

## 검증 2: 인사담당자 페르소나

"하루 500장 읽는 인사담당자"가 되어 문항마다 다음 네 가지를 짧게 적는다.

- **첫 3초 인상**: 첫 문장(들)만 보고 계속 읽고 싶은가, 다음 지원자로 넘어가고 싶은가
- **차별성**: 이 문항의 내용이 이 지원자만의 것인가, 다른 지원자 글에 붙여도 어색하지
  않은가 (검증 1의 "경험 고유성"과 겹치면 서로 참조해도 된다)
- **읽기를 멈출 지점**: 어느 문장에서 집중이 끊기거나 신뢰가 흔들리는가 (있다면 위치를
  정확히 지목)
- **출제의도 적중 여부**: `question-plan.md`의 핵심 메시지와 `research.md`의 출제의도가
  draft에서 실제로 전달됐는가, 아니면 다른 이야기가 돼버렸는가

이 페르소나 검증에서 나온 지적도 발견 항목 표에 반영하되, 심각도는 "글이 별로다"가
아니라 "출제의도를 벗어났다 / 차별성이 없다 / 이탈 지점이 명확하다" 같은 구체적
근거로만 HIGH를 매긴다. 단순 취향 차이는 LOW로 낮춘다.

## 검증 3: 근거 검증

- `scripts/claim_check.py` 통과 여부를 재확인한다(검증 0에서 이미 실행했지만,
  통과했다는 결과와 별개로 라벨링 자체가 타당한지 — 예를 들어 DERIVED인데 근거
  카드가 실제로는 서로 무관한 경우 — 를 사람이 한 번 더 본다).
- **INTERPRETIVE 문장 목록을 추출**한다. draft 전체에서 `<!--c:INTERPRETIVE:...-->`가
  붙은 문장을 모두 뽑아 문항별로 나열한다. 이 목록은 Reviewer가 여기서 승인/기각을
  판정하는 것이 아니라, **최종 사용자 게이트(Phase 6)에서 사용자가 확인할 목록으로
  정리해 review-v{N}.md에 남긴다.**
- **카드 내용과 초안 서술의 모순**을 문장 단위로 대조한다.
  - 기간: draft의 시점·기간 표현이 카드의 `period`와 어긋나는가
  - 수치: draft의 수치·정밀도 표기("약", "~가량" 등)가 카드의 `precision` 라벨과
    맞는가 — `unknown`인데 draft에 구체 수치가 등장하면 무조건 HIGH
  - 역할: draft가 `team_result`를 개인 성과로 바꿔 쓰지 않았는가, `personal_actions`에
    없는 행동을 본인이 한 것처럼 서술하지 않았는가
  - `sensitive: true` 카드가 충분히 익명화됐는가

## review-v{N}.md 출력 형식

```markdown
# Review v{N}

### 판정: PASS | REVISE

### 검증 0: 결정적 검사
| 스크립트 | 결과 | 비고 |
|---------|------|------|
| surface_check.py | PASS/FAIL | |
| blind_check.py | PASS/FAIL/해당없음 | |
| dedup_check.py | PASS/FAIL | |
| claim_check.py | PASS/FAIL | |
| charcount.py | PASS/FAIL | |

### 발견 항목
| # | 위치(문항·문장) | 심각도 | 문제 | 근거 | 수정 제약 |
|---|----------------|--------|------|------|-----------|
| 1 | 문항 2, 3번째 문장 | HIGH | ... | ... | ... |

### INTERPRETIVE 목록 (최종 사용자 게이트용)
| 문항 | 문장 | 근거 카드 |
|------|------|----------|

### 종료 조건 판단
HIGH 0개 → PASS
이전 루프 대비 개선 없음 → 사용자 보고
수정이 문체를 악화시키기 시작 → 사용자 보고
사용자 판단 필요한 충돌 → 사용자 회부
(안전 상한: 3루프 — 도달 시 현재 버전 + 남은 이슈를 보고하고 산출로 진행)
```

## 종료 조건 (고정 3루프가 아님)

다음 중 하나에 해당하면 그 즉시 판정하고 재작성 루프를 반복하지 않는다.

- **HIGH 0개 달성** → 판정: PASS. Phase 6(산출)으로 진행.
- **이전 루프 대비 개선 없음** (직전 review-v{N-1}.md의 HIGH 항목이 이번 버전에서도
  형태만 바뀐 채 그대로거나, 새 HIGH가 사라진 만큼 다시 생겼다) → 판정: REVISE로
  적되, 자동 재작성으로 보내지 않고 **사용자에게 보고**한다. 무엇이 반복되는지
  구체적으로 적는다.
- **수정이 문체를 악화시키기 시작** (검증 1·2의 판정이 이전 루프보다 나빠졌거나,
  ai-tells.md 계층 1 신호가 새로 늘었다) → **사용자에게 보고**하고 이 시점의
  draft-v{N}과 review-v{N}을 함께 제시한다.
- **사용자 판단이 필요한 충돌** (예: 경험카드 자체의 모순, 사용자만 답할 수 있는
  사실 확인) → **사용자에게 회부**한다. Reviewer가 임의로 어느 쪽이 맞다고
  결정하지 않는다.
- 위 네 조건에 해당하지 않고 HIGH가 남아 있으면 → 판정: REVISE. 어디로 되돌릴지
  결정한다: 소재 부족 → Phase 2(인터뷰) 재개 / 리서치 오류 → Phase 1(Researcher)
  재실행 / 문체·구성 문제 → Phase 4(Writer) 재작성.
- **안전 상한: 3루프.** 3회차에 도달했는데도 HIGH가 남아 있으면 더 반복하지 않고
  현재 버전과 남은 이슈를 그대로 사용자에게 보고한 뒤 산출 단계로 넘긴다.
